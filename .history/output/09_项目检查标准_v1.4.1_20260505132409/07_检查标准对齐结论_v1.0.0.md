# 检查标准对齐结论

> 检查对象：`output/09_项目检查标准_v1.4.0/`、`.agents/rules/`、`.agents/skills/` 与 `框架体检/scripts/framework-check.ps1`。检查时间：2026-05-04 23:20。

---

## 总结

09 子项目的大方向是对的：它已经覆盖了根目录六文件夹、`.agents` 目录清单、`output` 子项目文件夹命名、`.memory` 当前态结构、`.history` 三分区结构、`.temp` 与 `input` 存在性。

但它还没有完全对齐 `.agents` 的 rules 与 skills。主要问题是：09 标准偏“目录骨架检查”，而 rules/skills 已经进一步要求“子项目内部文件结构、内容文件编号、Skill 标准骨架、`.ps1` BOM、知识地图对齐、系统记录分流”等操作级约束。

---

## 已经对齐的部分

1. 根目录六文件夹要求对齐：`.agents/`、`.history/`、`.memory/`、`output/`、`input/`、`.temp/` 均被纳入标准点 #1。
2. `.agents/` 顶层清单大体对齐：4 个 rule 文件与 6 个 Skill 目录均被标准点 #2 覆盖。
3. `output/` 子项目文件夹命名对齐：`{编号}_{主题}_v{x.y.z}` 与 `filename-rules.md`、`子项目管理` Skill 的主规则一致。
4. `.memory/` 当前态结构对齐新规则：09 标准点 #4 采用“对话记录 / 系统记录 5 文件 / 知识提炼 / 全局知识地图”的结构，和 `memory-rules.md` 以及当前实际 `.memory/` 结构一致。
5. `.history/` 大结构对齐：09 标准点 #5 已覆盖 `.agents/`、`.memory/`、`output/` 三个历史分区，以及 `input/`、`.temp/` 不进历史的原则。

---

## 尚未对齐的部分

### 1. `output/` 子项目内部结构漏检

09 标准点 #3 只检查 `output/` 根目录下的文件夹命名和零散文件，但 rules/skills 还要求每个子项目内部必须有：

- `目录.md`
- `版本记录.md`
- 内容文件按 `{NN}_{主题}_v{x.y.z}.扩展名` 命名
- 新建或 MAJOR/MINOR 修改内容文件时同步维护 `目录.md`

依据来自 `direction-rules.md`、`filename-rules.md`、`version-control-rules.md`、`子项目管理/SKILL.md`。`framework-check.ps1` 也已经把“目录.md 完整性”和“文件编号”列为第 6、7 项检查。

本次运行框架体检的结果也证明了这个缺口：当前有 9 个旧子项目缺少 `目录.md`，17 个内容文件没有 `{NN}_` 前缀。也就是说，09 标准如果照现在执行，会漏掉框架体检已经判出的红项和黄项。

建议：在 09 中新增一个标准点，专门检查“子项目内部固定文件与内容文件编号”。

### 2. `框架体检` 实际检查项没有被 09 完整覆盖

`框架体检` Skill 的脚本实际检查 7 项：

- 对话记录完整性
- `.ps1` BOM
- `.memory` 当前区洁净度
- 全局知识地图对齐
- `input/` 路径污染
- `目录.md` 完整性
- 内容文件 `{NN}_` 编号

09 当前只覆盖其中一部分。尤其缺少 `.ps1` BOM、全局知识地图对齐、`input/` 路径污染、目录完整性、文件编号这些运行时健康检查。

建议：09 标准要么新增“框架体检运行项”标准点，要么在现有各标准点里补齐对应检查项。

### 3. Skill 骨架要求口径不统一

09 标准点 #2 写的是“每个 Skill 含 `SKILL.md`，特定 Skill 含 `scripts/`”。这能覆盖当前最低可运行结构，但没有覆盖 `创建技能` 与 `安装技能` references 中写明的标准骨架：

- `SKILL.md`
- `agents/openai.yaml`
- `references/`
- `scripts/`
- `assets/`

当前实际状态也不一致：`创建技能`、`安装技能`、`框架体检` 具备完整骨架；`子项目管理`、`版本控制备份`、`记忆管理` 只有 `SKILL.md` 与 `scripts/`。

建议：先决定项目级 Skill 标准到底采用“最小运行骨架”还是“完整薄 Skill 骨架”。09 标准点 #2 应按这个决策重写，避免把正常结构误报为多余，或把缺失骨架漏报。

### 4. 系统记录文件口径存在 rules/skills 冲突

09 标准点 #4 与 `memory-rules.md` 对齐，采用 `.memory/系统记录/` 下 5 个固定文件：

- `规则变更记录.md`
- `技能变更记录.md`
- `脚本治理记录.md`
- `教训库.md`
- `索引.md`

但 `子项目管理/SKILL.md` 和 `记忆管理/SKILL.md` 仍残留旧口径 `rules-skills.md`。`记忆管理/SKILL.md` 内部甚至同时出现“分类写 5 文件”和“统一写 rules-skills.md”两套说法。

当前实际 `.memory/系统记录/` 已经是 5 文件结构，所以这里更像是 Skill 文案落后于 rule，而不是 09 标准错。

建议：修正 `子项目管理` 与 `记忆管理` Skill 文案，统一为 5 文件 + `索引.md` 计数。

### 5. `.history/.agents` 当前态副本口径需要收敛

09 标准点 #5 对 `.history/.agents/skills/` 已写明“无时间戳 Skill 当前态副本禁止存在”，这与新版备份逻辑一致。但它对 `.history/.agents/rules/` 仍允许无时间戳当前态副本。

实际备份脚本的 `CONFIG` 模式只创建带时间戳的 rule 备份；`FOLDER` 模式也只创建带时间戳的 Skill 快照。当前 `.history/.agents/` 下仍有部分无时间戳残留目录或文件，需要明确它们到底是合法 current copy，还是历史迁移残留。

建议：统一为“`.agents` 历史只保留带时间戳快照/备份”；若仍要保留当前态副本，应在 backup 脚本和 rules 中明确写入。

---

## 建议修正顺序

1. 优先补 09 的 `output` 子项目内部检查：`目录.md`、`版本记录.md`、内容文件 `{NN}_` 编号。
2. 统一 `.memory/系统记录/` 口径：以 `memory-rules.md` 和当前实际结构为准，修正两个 Skill 里的 `rules-skills.md` 残留。
3. 决定 Skill 骨架标准：最小运行骨架或完整薄 Skill 骨架，之后再改 09 标准点 #2。
4. 把 `framework-check.ps1` 的 7 项检查映射进 09，避免“文档标准绿、脚本体检红”。
5. 收敛 `.history/.agents` 对无时间戳 current copy 的定义。

---

## 本次检查结论

**不完全对上。** 09 标准可以作为“第一层目录规范”，但还不能作为完整框架体检标准。若要让 09 成为权威检查标准，需要至少补齐 `output` 子项目内部结构、框架体检 7 项、Skill 骨架、系统记录口径、`.history/.agents` current copy 规则这五块。
