用户明确说明了整个项目的目标形态：通过 EXE 初始化项目骨架；日常提问触发 Skill 建立子项目、文档落盘、版本留存和 memory 记录；发现系统问题后进入 00 系统治理；系统治理需要有一份专门给维护骨架的 AI 使用的汇总文件；维护完成后同步到 05 工作区初始化工具并重新封装 EXE；用户在老项目中运行新版 EXE，再通过项目规范化 Skill 把老框架迁移到新框架。

## 0005 · 2026-05-05 · 维护总览图拆分首次剥离与后续循环

### 触发场景

用户指出 `01_项目骨架维护总览_v1.0.0.md` 中的循环图不准确：首次剥离/首次初始化不应和后续迭代混在同一个循环里，应该先把首次过程剥离出来，后续才进入循环。

### 修正方案

将原来的单环 Mermaid 图改为两层结构：

1. **首次剥离/初始化**：形成骨架思路，通过 05 工作区初始化工具封装首版 EXE，用户运行首版 EXE 创建标准工作区。
2. **后续迭代循环**：用户提问、子项目落盘、版本备份、记忆记录、系统治理、维护 AI 升级骨架、同步到 05 重新封装新版 EXE、老项目运行新版 EXE 注入新骨架、项目规范化 Skill 迁移旧状态，然后继续提问。

### 影响的文件

- `output/00_系统治理_v1.2.1/01_项目骨架维护总览_v1.0.0.md`
- `output/00_系统治理_v1.2.1/目录.md`
- `output/00_系统治理_v1.2.1/版本记录.md`
- `.memory/全局知识地图.md`
- `.memory/知识提炼/项目骨架演化机制.md`

### 教训

系统演化图要区分“一次性引导过程”和“可重复迭代过程”。把首次初始化放进循环，会误导后续维护 AI 以为每一轮都要重新执行首版骨架创建。
