# 记忆系统治理方案

> 目标：将研究型对话与系统治理型对话分流存储，避免 Rule、Skill、脚本、备份、记忆机制等系统层讨论混入具体业务子项目的对话记录。

---

## 1. 背景与问题

当前 `.memory/对话记录/{子项目文件夹名}.md` 默认承接了大部分会话摘要，但其中混入了以下不属于业务研究本身的内容：

- Rule 设计与修改
- Skill 设计与修改
- 脚本实现与修复
- 备份流程与 `.history` 机制
- `.memory` 结构与写入规则
- 自动化工作流与治理策略

这些内容属于“系统治理记忆”，不应继续沉入某个具体研究话题的子项目记录中，否则会带来三个问题：

- 子项目对话记录被系统维护内容污染，业务上下文不纯
- 后续回看某个研究项目时，难以快速定位真正与主题相关的讨论
- 系统层知识散落在多个项目文件中，后续维护 Rule / Skill / 脚本时难以集中追踪

---

## 2. 方案目标

本方案要实现以下目标：

1. 将对话记忆拆分为“项目记忆”和“系统治理记忆”两条线
2. 建立明确的路由判断规则，决定每轮对话写入何处
3. 保证业务研究内容仍进入子项目记录，系统治理内容集中归档
4. 允许一轮对话同时涉及两类内容时分别记录，避免混写
5. 为后续 Rule、Skill、脚本、记忆机制演进提供统一的系统记忆入口

---

## 3. 记忆分层设计

### 3.1 项目记忆

项目记忆用于记录与某个研究话题直接相关的内容。

存储位置：

```text
.memory/对话记录/{子项目文件夹名}.md
```

适用内容：

- 研究问题本身
- 业务分析过程
- 研究结论
- 输出文件形成过程
- 与该主题直接相关的讨论、澄清、修订

### 3.2 系统治理记忆

系统治理记忆用于记录与工作流、规则、技能、脚本、目录结构、备份机制、记忆机制相关的内容。

推荐存储位置：

```text
.memory/系统记录/
```

第一阶段建议先采用单文件聚合：

```text
.memory/系统记录/rules-skills.md
```

适用内容：

- Rule 的增删改
- Skill 的增删改
- 脚本逻辑设计、修复与重构
- `.history` 备份机制
- `.memory` 目录结构与触发规则
- 自动化工作流、路由逻辑、治理策略

---

## 4. 新目录结构

建议将 `.memory` 目录治理为如下结构：

```text
.memory/
├── 对话记录/
│   └── {子项目文件夹名}.md
├── 系统记录/
│   ├── rules-skills.md
│   ├── backup-workflow.md
│   └── memory-system.md
├── 知识提炼/
│   └── {子项目文件夹名}_v1.0.0.md
└── 全局知识地图_v1.1.0.md
```

说明：

- `对话记录/`：只放研究主题的项目对话记录
- `系统记录/`：只放系统治理层记忆
- `知识提炼/`：继续承接业务研究的结构化结论
- `全局知识地图`：继续维护子项目总索引

---

## 5. 路由规则设计

每次准备写入记忆时，先判断本轮对话属于哪一类。

### 5.1 三种路由结果

- `PROJECT`：写入项目对话记录
- `SYSTEM`：写入系统治理记录
- `BOTH`：两边都写，但摘要分开写

### 5.2 判定规则

若本轮对话主要讨论以下内容，判定为 `PROJECT`：

- 研究主题本身
- 面向主题的分析、推演、提炼
- 输出内容或结论

若本轮对话主要讨论以下内容，判定为 `SYSTEM`：

- Rule
- Skill
- 脚本机制
- 备份策略
- 目录规范
- Memory 系统设计
- 自动化工作流

若一轮对话同时涉及：

- 一部分是业务研究内容
- 一部分是系统治理调整

则判定为 `BOTH`，分别写入对应文件，禁止合并成一段。

---

## 6. 关键规则文本建议

建议将以下文本写入 `memory-rules.md`：

```md
对话记忆分为两类：

1. 项目对话记录：存储到 `.memory/对话记录/{子项目文件夹名}.md`
2. 系统治理记录：存储到 `.memory/系统记录/{主题文件}.md`

凡是讨论以下内容时，禁止写入子项目对话记录，必须写入系统治理记录：
- Rule
- Skill
- 脚本机制
- 备份策略
- 目录规范
- Memory 系统设计
- 自动化工作流

若同一轮对话同时包含项目内容与系统内容，允许双写，但必须分别摘要，禁止混写。
```

---

## 7. 对记忆管理 Skill 的改造建议

建议为 `记忆管理` Skill 增加“路由判断”步骤：

```md
### 第 0 步：判断记录路由
- 若本轮内容属于研究主题：写入项目对话记录
- 若本轮内容属于规则/技能/工作流治理：写入系统治理记录
- 若两者同时存在：分别追加到两个文件
```

还应补充以下约束：

- 结束回复前必须完成路由判断
- 若系统记录文件不存在，先初始化再追加
- 若是纯系统治理对话，禁止写入任何业务子项目记录

---

## 8. 对子项目管理 Skill 的影响

该方案生效后，`子项目管理` Skill 只负责：

1. 创建新的 `output/{主题}_{时间戳}` 子项目目录
2. 初始化 `.memory/对话记录/{子项目文件夹名}.md`
3. 更新 `全局知识地图`

它不再负责承接 Rule / Skill / 脚本治理相关的系统对话记录。

---

## 9. 第一阶段落地建议

为了降低复杂度，建议分两阶段实施。

### 第一阶段

先创建一个系统治理总文件：

```text
.memory/系统记录/rules-skills.md
```

优点：

- 实现简单
- 便于快速开始
- 避免过度拆分

### 第二阶段

当系统治理对话变多时，再拆成：

- `rules-skills.md`
- `backup-workflow.md`
- `memory-system.md`

这样便于后续专题化维护。

---

## 10. 迁移建议

对现有已经混入子项目记录的系统治理内容，建议做一次清理迁移：

1. 从各个 `.memory/对话记录/{子项目文件}.md` 中识别系统治理条目
2. 将这些条目迁移到 `.memory/系统记录/` 下的对应文件
3. 在原子项目记录中删除纯系统治理条目
4. 若某条记录同时含业务上下文，只保留与业务直接相关的部分

---

## 11. 最终结论

本方案的核心不是“再多建几个文件夹”，而是建立一套清晰的记忆边界：

- 业务研究内容归业务项目
- 系统治理内容归系统记录
- 混合内容拆分记录

这样可以同时提升：

- 子项目记录的纯度
- 系统知识的可维护性
- 后续 Rule / Skill / Script 演进时的可追踪性
