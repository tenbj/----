Knowledge Workspace / Operating Model

知识研究项目是怎样运作起来的

这套机制的核心不是“把文件放进几个目录”那么简单，而是把工作单元、内容安全、版本演进和长期记忆拆成四个彼此协作的系统。 子项目负责承接每个新主题，备份负责兜底，版本负责表达内容状态，记忆负责把上下文沉淀下来。

子项目 每个新话题先创建一个独立工作空间，所有产出都归档到自己的目录里。
备份 修改前先快照，确保旧内容能追溯，不让一次改动破坏历史。
版本 用 vMAJOR.MINOR.PATCH 表达变更幅度，让“改了什么”一眼可懂。
记忆 把项目对话、系统治理、知识提炼分流存储，避免长期上下文混乱。

四个系统，各管一件事

把这四层分开之后，项目才会稳定。否则新主题、旧版本、系统规则和对话摘要会互相污染，最后谁也说不清。

.agent

规则层与工具层。这里定义“应该怎么做”，例如子项目命名规则、备份规则、记忆路由规则。

output

工作层。每个子项目都是一个独立的输出空间，研究文档、HTML 页面、阶段成果都放这里。

.history

归档层。任何重要修改前先留快照，保证能回看旧文件、旧规则和旧技能目录。

.memory

记忆层。负责保存项目对话、系统治理记录、知识提炼和全局知识地图。

一条完整工作流

项目不是从“写文件”开始，而是从“先建立边界”开始。每个新主题都沿着下面这条链路往前走。

01

新话题进入

先判断是不是新的研究主题。如果是，就不能直接把文件丢进 output/ 根目录。

02

创建子项目

生成形如 03_项目运作机制_202605031830 的目录，让这个话题拥有自己的工作空间。

03

开始产出文件

所有 HTML、Markdown、分析稿都只写进这个子项目，不越界，不混放。

04

修改前先备份

内容文件走版本备份，规则文件走配置快照，Skill 目录走文件夹快照。

05

回复后沉淀记忆

系统自动判断这轮内容属于项目、系统还是两者兼有，然后分别写入对应记忆文件。

子项目机制

子项目是这套系统的最小工作单元。它不是一个普通文件夹，而是一次主题研究的边界。

命名规则

固定格式

output/{编号}_{主题}_{YYYYMMDDHHmm}

  • 编号表示顺序，例如 01、02、03
  • 主题必须是中文，直接说明研究对象
  • 时间戳标记创建时刻，避免重名，也让来源更清晰
当前项目实例

已经存在的子项目

  • 01_版本管理_202605031505
  • 02_记忆系统治理方案_202605031801
  • 03_项目运作机制_202605031830

备份与版本机制

备份解决“安全”，版本解决“表达”。两者不是一个概念，但总是成对出现。

FILE 模式

用于 output/ 或其他内容文件。旧文件会连同原版本号一起进入 .history/，当前工作文件再切到新版本号，所以子项目目录里只保留最新版本。

CONFIG 模式

用于 .agent/rules/。原文件名保持不变，只在 .history/.agent/rules/ 留一份时间戳快照。

FOLDER 模式

用于 .agent/skills/。不是只备份一个文件，而是把整个 Skill 目录快照到 .history/。

版本号语义

为什么要区分 MAJOR / MINOR / PATCH

  • MAJOR：结构重构、框架变化、旧内容不再兼容
  • MINOR：新增章节、新增说明、向下兼容的扩展
  • PATCH：修字句、补细节、纠正错误
典型结果

一个文件如何演进

例如当前页面首次创建为 项目运作机制说明_v1.0.0.html。以后继续修改时，旧版本会以原文件名进入 .history/output/...，工作区只留下新的 v1.1.0、v1.2.0 这类最新文件。

记忆机制

记忆系统的关键不是“都记下来”，而是“记对地方”，并且把当前态和历史态拆开。否则项目上下文、旧版本和系统治理会越积越乱。

路由规则

PROJECT / SYSTEM / BOTH

  • PROJECT：研究主题本身、输出内容、分析过程
  • SYSTEM：Rule、Skill、脚本、备份策略、记忆结构
  • BOTH：既有业务内容，也有系统治理内容，必须拆成两条分别写
存储位置

四类记忆文件

  • .memory/对话记录/：项目对话摘要，按子项目追加
  • .memory/系统记录/rules-skills.md：系统治理对话集中记录
  • .memory/知识提炼/：当前态使用稳定文件名，历史版本进入 .history/.memory/知识提炼/
  • .memory/全局知识地图.md：记录全部子项目索引，历史版本进入 .history/.memory/全局知识地图/

一句话抓重点

如果只记住四句话，这套系统也就抓住八成了。

  • 子项目是工作边界，新话题先建目录，再产出内容。
  • 备份是安全网，任何重要修改前都先留旧版本。
  • 版本号是语言，不只是文件名装饰，它在表达改动幅度。
  • 记忆必须分流，而且 `.memory` 只保留当前态，历史态统一进入 `.history/.memory/`。

这套机制最终解决了什么

它把一个容易越做越乱的知识工作区，变成了一个可以持续扩展、持续回看、持续演进的研究系统。

避免内容散落

每个主题都能找到自己的工作区，不会和别的话题混在一起。

避免改错无痕

任何一次变更都能追溯到旧状态，出错时能快速定位和恢复。

避免上下文污染

系统机制的讨论不再混入业务子项目，长期维护会轻很多。

适合怎么用这页？ 你可以把它当作项目总览页、给新协作者的 onboarding 页面，或者作为后续补充流程图与规则说明的基底版本。 以后只要机制有变化，就继续按版本规则迭代这份 HTML 即可。
