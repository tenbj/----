# 主控技能创建增量规则

这份文件只记录 `zqz_orchestrator` 在“创建 skill”时额外强调的本地约束，不替代通用 skill 设计常识。

## 什么时候读

- 要新建主控本地 runtime skill
- 要给工作区补新的本地 skill
- 要把厚 skill 改造成薄 skill
- 要判断某条规则该留在 `SKILL.md`、`references/`、`scripts/` 还是 `knowledge/`

## 主控里的默认判断

1. 先问这是不是主控经常重复执行的控制面动作。
2. 如果只是偶发冷规则、维护说明或一次性讨论，优先写进现有 skill 的 `references/`。
3. 如果它会在“新建工作区 / 改造工作区 / skill 迁移”这类治理动作里反复出现，才值得做成独立 skill。
4. 新建 skill 前先确认不会把主控 runtime 层重新做厚。

## 产出的新 skill 必须满足

1. 顶级目录名默认使用中文全称。
2. 必须补齐：
   - `SKILL.md`
   - `agents/openai.yaml`
   - `references/`
   - `scripts/`
   - `assets/`
3. `SKILL.md` 只保留薄入口，不写成长篇百科。
4. 长命令、参数表、兼容性说明、复杂异常处理默认下沉到 `references/`。
5. 可执行且高确定性的逻辑默认下沉到 `scripts/`。
6. 真正的模板、图片、二进制物料默认下沉到 `assets/`。
7. 不主动生成 README、CHANGELOG、INSTALLATION_GUIDE 之类的噪音文件。

## 主控与工作区的分层判断

- 主控架构、路由、治理约束：留在主控 skill 体系里
- 工作区稳定业务真相：下沉到工作区 `knowledge/`
- 只服务单个 skill 的局部执行细节：留在该 skill 的 `references/`

## 默认收口方式

当你拿不准某段内容该放哪时，按这个顺序判断：

1. 它是长期稳定的业务真相，还是某个 skill 的执行细节？
2. 它会被多个 skill 复用，还是只在当前 skill 触发时才需要？
3. 它是给模型阅读的，还是给脚本/输出直接消费的？

默认答案：

- 稳定真相 + 多 skill 共用：`knowledge/`
- 薄入口之外的展开说明：`references/`
- 高确定性执行逻辑：`scripts/`
- 模板与物料：`assets/`
