# 系统记录文件结构说明

> 版本：v1.0.0 | 创建时间：2026-05-04 19:42
> 用途：解释 .memory/系统记录/ 下 5 个文件的职责和区分标准

---

## 总览

| 文件 | 角色 | 一句话 |
|------|------|--------|
| ules-skills.md | 索引 | 不存正文，只告诉你"去哪个文件找什么" |
| ules-变更记录.md | Rule 变更流水 | 记录 direction/filename/memory/version-control-rules.md 被改过什么 |
| skills-变更记录.md | Skill 变更流水 | 记录 5 个 Skill 文件夹被创/改过什么 |
| 脚本治理记录.md | 脚本问题记录 | 脚本出了什么 bug、编码怎么坏的、编译怎么修好的 |
| 教训库.md | 跨领域教训 | 不管具体改了哪个文件，只说"以后别再这样干了" |

---

## 用具体场景区分4个文件

### 场景 A："AI 改了 direction-rules.md，加了一条规则"

→ 写 **rules-变更记录.md**

因为它是 Rule 文件的增删改。

---

### 场景 B："AI 更新了记忆管理 Skill 的检查清单"

→ 写 **skills-变更记录.md**

因为它是 Skill 文件的内容变更。

---

### 场景 C："backup.ps1 写出乱码文件名，排查发现 BOM 丢了"

→ 写 **脚本治理记录.md**

因为它是一个脚本的编码 bug + 修复过程。核心是"脚本层面出了什么问题、怎么修好的"。

但这件事也包含教训——**所以也写 教训库.md**。

---

### 场景 D："用户说'我让你设计，你直接改了代码，太快了'"

→ 只写 **教训库.md**

因为这里不涉及改任何具体 Rule/Skill/脚本，但产生了一条跨领域的纠偏规则（"用户说设计/方案/讨论时，先出方案再动手"）。

---

### 场景 E："AI 新建了备份 Skill + 写了 backup.ps1 + 更新了 version-control-rules.md"

→ 三个文件都要写：
- skills-变更记录.md（新建备份 Skill）
- 脚本治理记录.md（新建 backup.ps1）
- rules-变更记录.md（更新 version-control-rules.md）

---

## 为什么看起来差不多

因为原来 553 行的 ules-skills.md 是**按时间不分青红皂白全部追加**的。一条记录里可能同时涉及 Rule 变更、脚本编码问题和跨领域教训。上一轮拆分的做法是用关键词匹配做自动分类——把同一件事的 Rule 层、Skill 层、脚本层、教训层拆到了不同文件。

所以同一个事件会出现"这 4 个文件里都有相关内容"的情况，看起来就像同一批东西分了 4 遍。

**正确的区分标准不是"这是哪个事件"，而是"这是哪个维度"**：

| 维度 | 问什么 | 对应文件 |
|------|--------|---------|
| Rule | 哪个 rule 文件改了什么？ | rules-变更记录 |
| Skill | 哪个 skill 增删改了什么？ | skills-变更记录 |
| 脚本 | 哪个脚本出了什么 bug？怎么修好的？ | 脚本治理记录 |
| 教训 | 从这次踩坑里学到了什么经验/纠偏规则？ | 教训库 |

同一个事件可以触达多个维度。上面场景 C（BOM 编码事故）就是典型的跨 3 个维度。
