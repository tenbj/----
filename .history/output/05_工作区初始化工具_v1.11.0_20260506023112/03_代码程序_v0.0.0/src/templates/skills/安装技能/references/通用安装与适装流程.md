# 通用安装与适装流程

## 先评估，再安装

只要已经出现明确安装意图，默认先做适装评估。

典型触发：

- 点名要安装某个 skill
- 给出候选 skill 列表，希望推荐
- 问“这个 skill 值不值得装”
- 准备批量安装

如果只是浏览目录或让你列清单，可以先列清单；一旦进入安装决策，就应切换到评估。

## 适装评估的 6 项默认维度

1. 新增知识价值
2. 能力重叠度
3. 运行环境耦合度
4. 流程负担
5. 触发范围
6. 上下文成本

## 动态判断原则

- 同一个 skill，在不同运行面下价值可能不同
- 同一个流程型 skill，在规则很薄的项目里可能补短板，在规则已完整的项目里可能只是加负担
- 如果用户只打算在单一平台用，可以适度放宽耦合度
- 如果用户要求跨平台复用，就必须严格看平台耦合风险

## 建议输出格式

```markdown
技能适装评估
- 结论：推荐安装 / 条件安装 / 不建议安装
- 建议范围：项目级试装 / 当前项目正式安装 / 暂不安装
- 新增知识价值：
- 能力重叠度：
- 运行环境耦合度：
- 流程负担：
- 触发范围：
- 上下文成本：
- 简短结论：
```

## 默认决策口径

- 推荐安装：新增价值明显，耦合可控，适合当前项目
- 条件安装：有价值，但有明显代价，默认先项目级试装
- 不建议安装：主要是重复已有能力，或明显拖慢日常任务
