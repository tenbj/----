# 主控适装评估与本地化规则

这份文件记录 `zqz_orchestrator` 在“安装或迁入外部 skill”时额外强调的判断和收口方式。

## 什么时候读

- 用户想安装某个外部 skill
- 主控想把旧 skill 迁入当前项目
- 工作区改造时发现需要顺手引入外部 skill
- 需要判断某个能力更适合留在全局层，还是当前项目本地层

## 先评估，再安装

只要已经出现明确安装意图，默认先做适装评估，不直接开装。

评估至少覆盖这 6 项：

1. 新增知识价值：它是否真的补充了当前模型和现有 skill 不稳定掌握的领域知识、工具集成或强确定性流程。
2. 能力重叠度：它是不是只把现有能力又包装了一遍。
3. 运行环境耦合度：它是否强依赖单一平台的专属机制。
4. 流程负担：它会不会让简单任务也被迫走重流程。
5. 触发范围：它会不会广泛干扰日常任务。
6. 上下文成本：它的常驻说明是否对得起所占上下文。

## 主控里的默认安装层级

- 通用能力：优先留在全局层；但这次主控已明确把 `创建技能` 和 `安装技能` 自己收为项目级 runtime skill。
- 主控控制面独有能力：允许项目级安装。
- 只影响一个现有 skill 的补充说明：优先写进那个 skill 的 `references/`。

## 安装后的必做本地化

1. 给迁入 skill 选一个中文目录名；脚本层默认建议显式传 `--name`。
2. 保留原始 `SKILL.md` 到 `references/原始技能说明.md`。
3. 重新生成薄 `SKILL.md`，只保留触发、固定动作、references 路由和边界。
4. 重新生成中文 `agents/openai.yaml`。
5. 补齐 `agents/`、`references/`、`scripts/`、`assets/` 四类目录。
6. 清理根目录噪音文件，避免 README、CHANGELOG 一类文档继续漂在热路径。

## 工作区建骨架或重构时的隐式触发

如果主控正在做下面这些动作，也应自然命中“安装技能”：

- 新工作区接入时发现需要迁入一个成熟外部 skill
- 现成目录改造时发现原地有旧 skill 需要本地化
- 技能盘点时发现某个外部 skill 值得项目级试装
- 新建工作区骨架后，需要把一个外部 skill 变成薄的本地入口

## 建议输出格式

适装评估的结论默认落成：

- 结论：推荐安装 / 条件安装 / 不建议安装
- 建议范围：项目级试装 / 当前项目正式安装 / 暂不安装
- 六项评估要点
- 简短结论
