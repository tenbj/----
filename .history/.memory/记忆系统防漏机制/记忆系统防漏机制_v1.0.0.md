# 知识提炼 · 记忆系统防漏机制

> 从研究过程中提炼的核心结论与知识点，持续更新。

---

## 2026-05-04 · 记忆记录遗漏的根因与防护

**核心观点**：
- 记忆系统有三层输出（对话记录、系统记录、知识提炼），缺一不可。但实际执行中，对话记录和知识提炼最容易被跳过。
- 根因不在于"规则不够"，而在于"规则被执行者当成了可选项"——记忆记录被当作"任务完成后的额外步骤"，而非回复本身的一部分。
- 防护不能只靠纪律，必须从规则文本和执行检查清单两个层面同时加固。

**关键框架/模型**：

### 三道防线

| 防线 | 位置 | 机制 |
|------|------|------|
| 第一道 | `memory-rules.md` 顶部 | 强制条款：「没有记录 = 回复没完」 |
| 第二道 | `记忆管理 Skill` 检查清单第 0 项 | 自检：「若正阅读此清单但记忆未写入→立即写入再发回复」 |
| 第三道 | 同上，第 6 项 | 数量自检：「对话记录文件数 ≥ 子项目文件夹数」 |

### 路由 BOTH 的陷阱

- 涉及 output/ 文件增删改 + 系统机制变更的项目，必须判 BOTH，不能只判 SYSTEM
- 不要以"这不只是补丁/延续"为由跳过 PROJECT 侧记录
- 判断依据：看在哪个子项目文件夹里做，而非"改的是什么"

### 知识提炼的触发缺口

- `memory-rules.md` 写「每次创建或修改 output/ 文件后必须追加知识提炼」
- `记忆管理 Skill` 写知识提炼是「手动触发」
- 二者矛盾给了跳过执行的台阶

**与其他话题的关联**：
- → 项目运作机制：四层记忆的联动链路本身就容易在"对话记录"和"知识提炼"两个节点断链
- → 记忆系统治理方案：分流机制只管"写到哪"，不管"写了没"，缺执行保障层
