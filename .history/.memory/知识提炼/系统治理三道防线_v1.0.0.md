# 知识提炼 · 系统治理三道防线

> 从系统治理方案中提炼的长期规则，用于后续 Rule、Skill、脚本改造时复用。

---

## 2026-05-07 · 研究来源、核心骨架、对话路由三道防线

**核心观点**：
- 系统治理不能只靠 AI 自觉，必须把关键风险做成硬防线。
- 研究可信度的关键不是“看起来权威”，而是每条核心资料都可访问、可验证或可下载，并说明当前仍权威的依据。
- 核心骨架需要授权门：未获用户明确允许时，AI 只能读、分析、出方案，不能修改 Rule、Skill、脚本、标准和 05 初始化模板。
- 对话处理需要先分类再执行：先判 `PROJECT / SYSTEM / BOTH`，再按任务类型选择 Rule、Skill、输出位置和记忆位置。

**关键框架/模型**：
- 来源核验防线：资料卡片、访问日期、可验证标识、当前权威性依据、证据等级。
- 核心骨架授权防线：核心路径清单、明确授权语义、修改前清单、备份与验证。
- 对话路由防线：内容路由 + 任务类型的二维分类表，固定到输出、备份、目录、记忆流程。

**与其他话题的关联**：
- → 项目骨架演化机制：本防线属于骨架自我迭代流程的约束层。
- → 记忆系统防漏机制：对话路由防线补强了“应该写到哪里”的前置判断。
- → PowerShell 编码陷阱：核心脚本修改仍需遵守 `.ps1` UTF-8 with BOM 防线。
