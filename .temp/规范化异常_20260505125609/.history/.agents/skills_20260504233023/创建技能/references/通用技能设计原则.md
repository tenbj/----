# 通用技能设计原则

这份文件保留“创建技能”作为元技能时最该反复调用的通用设计知识。

## 技能到底提供什么

一个好 skill 往往在做下面几件事中的一件或几件：

- 提供一个稳定工作流
- 提供模型本来不稳定掌握的领域知识
- 提供工具或文件格式的可靠集成方式
- 提供脚本、模板、参考资料等可重复复用的资源

## 设计时的三条总原则

### 1. 上下文是公共预算

- 默认先相信模型已经会普通常识和普通编码
- 只补它不稳定掌握的知识、约束和流程
- 每加一段说明前都问一句：这段信息是“必要知识”，还是“只是写起来安心”

### 2. 给足够的护栏，不给多余的束缚

- 高自由度：多个方法都可行，主要靠原则判断
- 中自由度：有推荐模式，但允许根据上下文调整
- 低自由度：脆弱、高风险或必须稳定复用的流程，应优先下沉到脚本或非常明确的步骤

### 3. 先问“该不该做成 skill”

不要只看“能不能做”，还要看“做成 skill 之后是不是在补能力”。

默认至少做这 6 项判断：

1. 新增知识价值
2. 能力重叠度
3. 运行环境耦合度
4. 流程负担
5. 触发范围
6. 上下文成本

## 适合优先做成 skill 的类型

- 知识型：补真实领域知识、私有规则、专有工具用法
- 强确定性型：把高风险流程固化成脚本或明确步骤
- 低耦合型：不死绑在某一个平台独占特性上

## 慎做成 skill 的类型

- 只是把“先查一下”“先规划一下”“注意安全”又写一遍
- 强依赖某平台专属机制，却没给替代路径
- 会让简单任务默认变得更慢、更重

## 元技能的额外要求

如果这个 skill 本身会创建其他 skill，它更应该遵守薄入口原则。否则它会把“厚 skill”继续复制出去。
