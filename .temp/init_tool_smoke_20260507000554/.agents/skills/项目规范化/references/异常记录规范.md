# 异常记录规范

## 记录位置

所有严格规范化中遇到的异常情况，都记录到当前 live `00_系统治理_v*` 子项目。

首选位置：

```text
output/00_系统治理_v*/02_课题研究_v1.0.0/{NN}_规范化异常记录.md
```

如果 `00_系统治理` 尚未三分类，先把 `00_系统治理` 自身迁为三分类结构，再写入异常记录。只有在迁移 `00_系统治理` 本身失败时，才临时写入根目录，并在异常中标注“待迁移”。

## 什么时候必须记录

- 脚本检查通过，但大模型发现与标准不一致
- 平铺子项目无法确定内容归类
- `.history` 中存在来源不明、命名不明或不应继续编辑的历史项
- `.memory` 中存在乱码、重复、错位、缺失且无法安全重建
- `.system/standards` 与实际脚本或 Skill 行为冲突
- 大模型做了标准推断，而不是直接来自 SSOT 明文
- 本次规范化留下了暂不处理的 WARN/FAIL

## 记录格式

```markdown
## {YYYY-MM-DD HH:mm} · {异常标题}

**发现位置**：`{路径}`
**对应标准**：`{标准文件或字段}`
**异常描述**：{哪里不一致}
**处理判断**：{已修复 / 已迁移 / 待用户确认 / 暂不处理}
**本次动作**：{执行了什么}
**后续建议**：{如有}

---
```

## 异常处理原则

- 能确定的，先备份再修复。
- 能迁移的，迁到标准位置。
- 无法判断归属的，记录后放入 `.temp/规范化异常_{yyyyMMddHHmmss}/`，保留原路径结构。
- 标准缺口不靠隐性习惯填补，必须写明推断依据。
- 不把异常只写进终端输出；必须落到 `00_系统治理`。
