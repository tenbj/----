# 版本管理规则（Semantic Versioning 中文指南）

## 1. 版本号结构

版本号采用三段式：

```
MAJOR.MINOR.PATCH
```

* **MAJOR（大版本）**：当进行不兼容的修改时。
* **MINOR（小版本）**：当新增向下兼容的功能时。
* **PATCH（补丁版本）**：当修复向下兼容的 bug 或做轻微改动时。

示例：

```
2.4.1
```

* 大版本 `2` → 第二代产品，可能有破坏性变化
* 小版本 `4` → 新增功能，向下兼容
* 补丁 `1` → 修复 bug

## 2. 版本号升级规则

| 场景           | 升级版本     | 描述                       |
| ------------ | -------- | ------------------------ |
| 向后不兼容的重大修改   | MAJOR +1 | API 改动、核心架构重构、功能删除或不兼容修改 |
| 向下兼容的新功能     | MINOR +1 | 新增功能、性能优化、新模块增加，但不影响现有功能 |
| 向下兼容的 bug 修复 | PATCH +1 | 修复 bug、安全漏洞、文档修改、轻微优化    |

**注意事项**：

* 每次大版本升级后，小版本和补丁号归零。例如：`1.5.3` → `2.0.0`
* 每次小版本升级后，补丁号归零。例如：`1.3.2` → `1.4.0`
* 修复 bug 时只增加 PATCH。例如：`1.4.2` → `1.4.3`

## 3. 特殊标记（可选）

* **预发布版本**：用于测试或 Beta 版本

```
1.0.0-alpha
1.0.0-beta.1
```

* **构建元数据**：用于内部标记或 CI/CD

```
1.0.0+20260503
```

## 4. 实际应用示例

1. 修复严重 bug：`1.2.3` → `1.2.4`
2. 新增功能：`1.2.4` → `1.3.0`
3. 架构大改，不兼容：`1.3.0` → `2.0.0`

## 5. 建议

* 保持语义化版本一致性，尤其大版本升级需慎重。
* 对外发布版本时写清 Release Note，说明变动内容。
* 内部迭代可加构建号或日期标记，例如：`1.3.0+build.20260503`.
