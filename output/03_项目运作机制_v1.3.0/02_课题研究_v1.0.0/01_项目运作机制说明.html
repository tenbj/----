知识研究工作区 · 运作机制 v1.3.0

这套项目如何把研究、规则、备份和记忆串起来

当前机制的核心是先建立边界，再产出内容：新话题先归属到子项目，子项目内部再按问题答疑、课题研究、代码程序三类分区。备份保证能回看旧状态，版本号表达变化幅度，记忆系统把每次决策沉淀到正确的位置。

更新日期：2026-05-05 当前子项目：03_项目运作机制_v1.3.0 内部结构：三分类子文件夹

四个系统，各管一件事

工作区不靠单个文件夹维持秩序，而是由输出、备份、规则技能和记忆四层共同协作。

output

承接所有正式产出。每个主题都是一个带版本号的子项目，内部按三类子文件夹组织。

.agents

存放 Rules 与 Skills。Rules 定义边界，Skills 定义可执行流程，脚本提供稳定操作入口。

.history

保存修改前快照。output 子项目、rules、skills、memory 当前态都有各自的备份模式。

.memory

保存对话摘要、系统治理记录、知识提炼和全局知识地图，让长期上下文不散落。

新话题从哪里开始

知识性回答不能直接从聊天框开始。先确认归属，再决定写入哪个子项目。

话题归属

回答前扫描 output/，判断问题属于已有子项目还是新话题。不存在没有子项目的知识性回答。

创建或定位子项目

新话题使用 子项目管理/scripts/new_project.ps1 创建 {编号}_{主题}_v1.0.0；旧话题继续在已有子项目内工作。

进入正确分类

普通问答进 01_问题答疑_v*，结构化研究进 02_课题研究_v*，代码与程序资产进 03_代码程序_v*。

落盘、更新目录、写记忆

文件落盘后更新根目录 目录.md，最终按 PROJECT / SYSTEM / BOTH 路由写入记忆。

子项目内部三分类

新子项目不再把文件平铺在根目录。根目录只保留目录与版本记录，内容进入三个分类文件夹。

01_问题答疑_v*

承接短问答、解释型回答、对已有研究的小范围补充。空文件夹为 v0.0.0，首次有内容后从 v1.0.0 起步。

02_课题研究_v*

承接完整研究文档，遵循问题、第一性原理、合理推导、结论、参考文献的结构。

03_代码程序_v*

承接脚本、源码、构建产物说明和程序资产。代码文件本身不强行套研究文档格式。

命名规则：分类文件夹带版本号，内部文件不带 _v*。例如 02_课题研究_v1.0.0/01_项目运作机制说明.html。

版本与备份

备份解决安全，版本号表达变化。每次改动前先备份，是这套系统的硬门槛。

对象 模式 结果
output/ 子项目 PROJECT 整个子项目快照进 .history/output/，live 文件夹 bump 版本号。
.agents/rules/ CONFIG 原文件名不变，在 .history/.agents/rules/ 留时间戳备份。
.agents/skills/ FOLDER 整个 Skill 文件夹快照，原 Skill 文件夹继续作为当前态。
.memory/知识提炼/ 与 全局知识地图.md MEMORY 当前文件名保持稳定，历史版本进入 .history/.memory/。
PowerShell 脚本额外约束：.agents 下的 .ps1 必须保持 UTF-8 with BOM，否则 Windows PowerShell 5.1 可能把中文读成乱码。

记忆路由

记忆不是越多越好，而是必须写到正确位置。项目内容与系统治理不能混在同一条记录里。

PROJECT

研究主题、业务分析、输出文件内容，写入 .memory/对话记录/{子项目文件夹名}.md。

SYSTEM

Rule、Skill、脚本、备份策略和目录规范，写入 .memory/系统记录/ 对应分类。

BOTH

同时包含项目内容与系统机制时，两边分别记录，禁止合并成一条摘要。

体检与规范化

框架体检只读，规范化可修复。两者的边界不同，不能混用。

框架体检

运行 .agents/skills/框架体检/scripts/framework-check.ps1。它检查对话记录、BOM、memory 洁净度、知识地图、input 污染、三分类结构和内部文件命名，只报告问题。

项目规范化

运行 .agents/skills/项目规范化/scripts/normalize.ps1 -Check 或 -Fix。修复前会先快照子项目；新结构或空项目可补齐三分类，老平铺项目不会自动搬迁内容。

一句话抓重点

用这几条判断一次操作是否在框架内。

  • 知识性回答先做话题归属，再进入对应子项目。
  • 子项目根目录只放 目录.md、版本记录.md 和三个分类文件夹。
  • 修改前先备份，output 子项目必须整项目快照。
  • 内部内容文件不带版本号，版本号由子项目和分类文件夹表达。
  • 系统治理记录写入 .memory/系统记录/，不要混进业务子项目对话记录。
本页是项目运作机制的当前态说明。后续 Rule、Skill、脚本或目录结构继续变化时，先备份 03 子项目，再迭代本页与目录、版本记录。
