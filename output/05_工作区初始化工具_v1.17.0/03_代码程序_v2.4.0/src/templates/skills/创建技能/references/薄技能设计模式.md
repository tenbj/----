# 薄技能设计模式

这份文件专门回答一个问题：怎么让 `SKILL.md` 变薄，但 skill 总能力不缩水。

## 薄的是入口，不是知识总量

正确的薄 skill 应该是：

- `SKILL.md` 只保留触发、固定动作、references 路由和边界
- 详细规则拆到 `references/`
- 高确定性逻辑拆到 `scripts/`
- 模板或物料放进 `assets/`

错误的薄 skill 是：

- `SKILL.md` 变薄了
- 但 references 没补回来
- 结果 skill 的总知识量一起缩水

## 标准分层

### `SKILL.md`

保留：

- 何时触发
- 这项 skill 解决什么问题
- 固定动作骨架
- 什么时候再读 references
- 边界和禁止事项

不要长期堆：

- 大段参数表
- 长篇命令示例
- 复杂异常排查手册
- 稳定业务真相原文

### `references/`

放只在触发这个 skill 时才需要展开的说明，例如：

- 参数表
- 长命令
- 平台兼容性说明
- 复杂样例
- 结果结构解释
- 缓存、断点续跑、复核策略

### `scripts/`

放：

- 高确定性执行逻辑
- 反复重写很浪费的代码
- 需要直接跑的工具脚本

### `assets/`

放：

- 模板
- 图片
- 字体
- 二进制样板

## 常见拆分模式

### 模式 1：入口 + 单主题参考

- `SKILL.md` 保留工作流骨架
- 一个 `references/*.md` 承接具体技术说明

### 模式 2：入口 + 多专题参考

适合同时有命令、策略、异常维护、结果结构说明的 skill。

例如：

- `references/cli.md`
- `references/strategy.md`
- `references/troubleshooting.md`

### 模式 3：入口 + 工作区知识

如果某些内容已经是工作区级稳定真相，应下沉到工作区 `knowledge/`，而不是留在 skill 私有 references 里。

## 反模式

- 只把 `SKILL.md` 砍短，却不补 references
- 同一条规则同时出现在 `AGENTS.md`、`SKILL.md`、`references/`
- 把稳定业务知识长期堆在 skill 私有 references 里漂移
