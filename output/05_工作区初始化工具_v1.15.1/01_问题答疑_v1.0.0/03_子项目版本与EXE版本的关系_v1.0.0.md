# 子项目版本与 EXE 版本的关系

> 子项目：05_工作区初始化工具  
> 落盘时间：2026-05-06  
> 类型：问题答疑

---

## 问题

为什么子项目的版本号（如 `v1.12.0`）和 EXE 的版本号（如 `v2.0.0`）不一样？

---

## 结论（一句话）

两个版本号追踪的是**不同维度的事物**，不要求同步，也不应该强制对齐。

---

## 两个版本号的语义

### 子项目版本号：`05_工作区初始化工具_vX.Y.Z`

- **追踪对象**：这个"项目容器"本身的内容变更历史
- **升级触发**：子项目文件夹下任何内容发生增删改时（包括 QA 文档、研究文档、源码、EXE 产物）
- **管理者**：`backup.ps1` 的 PROJECT 模式自动 bump
- **语义**：第 N 次对这个子项目做了修改

例：
- v1.8.0 → 新增了一致性检查报告（QA 文档新增）
- v1.9.0 → 引入 SSOT 标准架构分析（新文档）
- v1.10.0 → init_workspace.py 代码重写
- v1.12.0 → 全量同步模板并重新编译 EXE

### EXE 版本号：`初始化工作区_vX.Y.Z`（= `SKELETON_VERSION`）

- **追踪对象**：EXE 内置的**工作区骨架规格**版本
- **升级触发**：当 EXE 能够初始化出的工作区架构发生变化时（目录结构、必备文件、标准内容）
- **管理者**：`init_workspace.py` 中的 `SKELETON_VERSION` 常量手动维护
- **语义**：这个 EXE 实现的是第 N 版骨架规格

例：
- v1.6.0 → 支持老项目升级（先备份再替换）
- v1.7.0 → 加入 `课题研究` Skill
- v1.7.1 → 补入 `.temp/` 目录创建
- v1.9.0 → 加入 `.system/standards/` 初始化（SSOT 架构）
- v2.0.0 → 全量对齐最新规范（standards + 8个Skill + 文件命名规则）

---

## 为什么不要求对齐？

| 维度 | 子项目版本 | EXE版本 |
|------|-----------|--------|
| 含义 | "修改了几次这个项目" | "骨架规格演进到哪一代" |
| 升级触发 | 任何内容变更 | 骨架架构设计变更 |
| 步长 | 频繁（每次操作） | 较少（架构变化才升） |
| 管理工具 | backup.ps1 自动 | SKELETON_VERSION 手动 |

一次对子项目的修改可能：
- **不升 EXE 版本**：只是新增 QA 文档，EXE 逻辑没变
- **大幅升 EXE 版本**：骨架规格从 1.x 跳到 2.0（跨代升级）

强制两者对齐会导致：
- EXE 版本被大量无关的 QA 文档变更拉着走，失去语义
- 子项目版本被骨架规格大版本跳跃拉着走，丢失历史连续性

---

## 各自的大/小版本升级规则

两者都遵循 `MAJOR.MINOR.PATCH` 语义化版本格式，但升级触发条件不同。

### 子项目版本：`05_工作区初始化工具_vX.Y.Z`

> 来源：[filename-rules.md](file:///e:\my_project\知识研究\.agents\rules\filename-rules.md) §5 · [版本控制备份](file:///e:\my_project\知识研究\.agents\skills\版本控制备份\SKILL.md) PROJECT 模式

| 版本位 | 触发条件 | 典型场景 |
|--------|---------|---------|
| **MAJOR +1** | 内容结构、框架发生**不兼容的重大变化** | 子项目分类体系重组（如把 `src/` 从根目录迁入 `03_代码程序_v0.0.0/`） |
| **MINOR +1** | **新增内容/章节**，向下兼容 | 新增一篇问题答疑、新增课题研究、同步 Skill 模板并重新编译 EXE |
| **PATCH +1** | 描述微调、修复错误、轻微补充 | 修正错别字、补充一句话说明、修复文档格式 |

**管理者**：`backup.ps1` PROJECT 模式自动 bump。AI 每次修改子项目文件前执行备份，按本次变更幅度选择 `-ChangeType MAJOR|MINOR|PATCH`。

### EXE 版本：`初始化工作区_vX.Y.Z`（= `SKELETON_VERSION`）

> 来源：[init_workspace.py](file:///e:\my_project\知识研究\output\05_工作区初始化工具_v1.11.0\03_代码程序_v0.0.0\src\init_workspace.py) 顶部 `SKELETON_VERSION` 常量

| 版本位 | 触发条件 | 典型场景 |
|--------|---------|---------|
| **MAJOR +1** | 骨架规格发生**不兼容的重大变化** | v1.x → v2.0.0：全量对齐最新规范（standards + 8 个 Skill + 文件命名规则全部重做） |
| **MINOR +1** | 骨架**新增能力**，向下兼容 | v1.6.0 → v1.7.0：加入 `课题研究` Skill；v1.8.0 → v1.9.0：加入 `.system/standards/` 初始化 |
| **PATCH +1** | 骨架**修复/补漏**，向下兼容 | v1.7.0 → v1.7.1：补入 `.temp/` 目录创建；v1.7.1 → v1.7.2：修正 Skill 判定逻辑 |

**管理者**：`init_workspace.py` 中 `SKELETON_VERSION` 常量，由开发者**手动**修改。`build.ps1` 中的 `$Version` 需同步。

### 对比速查

| | 子项目版本 | EXE 版本 |
|---|---|---|
| 加 **MAJOR** | 子项目文件夹内部**结构重组** | 骨架规格**不兼容重构** |
| 加 **MINOR** | 子项目内**新增任何内容** | 骨架**新增能力** |
| 加 **PATCH** | 子项目内**微调/修复** | 骨架**补漏/修 bug** |
| 谁触发 | AI 改文件前自动 bump | 开发者手动改常量 |

### 为什么会同时看到 v1.11.0（子项目）vs v2.0.0（EXE）

一次对子项目的修改可以同时触发两种不同幅度的版本变更：

| 操作 | 子项目 | EXE |
|------|--------|-----|
| 只新增一篇 QA 文档 | +MINOR（新内容） | 不升（骨架没变） |
| 全量同步 Skill 模板 + 骨架规格全量重构 | +MINOR（新内容） | +MAJOR（不兼容重构） |

两者各自独立判断变更幅度，不要求对齐。

## 类比

> 把「子项目」类比为一本**研究笔记本**，把「EXE」类比为笔记本里夹着的一张**规格说明书**。
>
> 笔记本已经写到第 12 章（v1.12.0），但里面最新的规格说明书版本是 2.0（v2.0.0）。
> 章节数和说明书版本号没有必然关联，各自表达不同的演进轨迹。
