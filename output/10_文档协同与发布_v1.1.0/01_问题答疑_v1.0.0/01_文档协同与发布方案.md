# 文档协同与发布方案

针对本地基于 Markdown 文件的知识体系，要实现“与他人协同”以及“让产出更容易被看到”，可以采用以下两套结合的方案：基于 Git 的异步协同，以及基于静态站点生成（SSG）的公开发布。

## 一、 与他人协同编撰（协作层）

Markdown 加上这套基于纯文本的目录规范，天然非常适合使用标准的程序员工作流进行协同。

### 1. Git + 远程仓库 (如 GitHub/GitLab/Gitee)
这是最推荐的强协同方案：
- **原理**：将整个工作区（`output/`、`.memory/`、`.agents/` 等）作为一个 Git 仓库。
- **协同方式**：其他协作者 clone 这个仓库，在他们本地也可以通过同样的 Trae/VS Code + Agent 的方式进行查阅和编辑。如果他们修改了内容，可以通过 Pull Request (PR) 或直接 Push 提交。
- **兼容性**：你目前的版本控制备份脚本（`.history`）与 Git 互不冲突。`.history` 记录了 Agent 的改动轨迹，而 Git 则记录了人类与他人协作的轨迹。

### 2. 基于 Obsidian + 云同步 (轻量化协同)
如果不希望使用复杂的 Git 流程，可以使用 Obsidian：
- **原理**：将整个工作区根目录作为一个 Obsidian Vault（知识库）打开。
- **协同方式**：利用坚果云、Syncthing 等网盘同步工具，将整个目录在小团队内同步。大家都可以用 Obsidian 打开该目录直接阅读双链和文件，利用 Obsidian 丰富的插件生态（如评论、高亮插件）进行审阅协同。

---

## 二、 让产出更容易被看到（发布层）

如果只为了“让别人容易看到（阅读）”而不要求对方修改，最好的方式是将 `output/` 里的结构化 Markdown 转换成美观的在线网页。

### 1. Material for MkDocs（强烈推荐）
MkDocs 是一个快速、简单、极其适合这种带有编号目录结构的静态网站生成器。
- **优势**：它对 Markdown 原生支持极好，主题（Material）非常现代美观，支持全局搜索、深浅色模式切换。
- **实施方法**：
  1. 在根目录配置一个简单的 `mkdocs.yml`，把 `output/` 设置为 `docs_dir`。
  2. 运行 `mkdocs build`，它会一键生成所有 HTML 静态文件。
  3. 利用 GitHub Pages / Gitee Pages 或者任意 Nginx 服务器直接免费托管。别人通过网址就能看到结构清晰的左侧导航树和内容。

### 2. Docusaurus 或 VitePress
如果你的知识库越来越像一个产品文档或者博客，可以使用这两款工具。
- **优势**：支持更加丰富的自定义组件、更好的性能体验（SPA 单页应用），支持在 Markdown 中嵌套更复杂的互动能力。
- **适用场景**：当你想把某个具体的子项目（例如 `06_睡眠优化`）包装成一个体系化的小教程对外发布时。

### 3. 导出为传统格式 (PDF / Word)
当面对传统阅读习惯的受众时：
- **工具**：使用 [Pandoc](https://pandoc.org/) 或 Obsidian 的导出插件。
- **方法**：将某一个课题研究目录下的 Markdown 拼合，一键转换为带目录的 PDF 或 Word 文档，发送给他人。

## 总结建议

- **内部小团队协作**：直接把项目扔上 Git 私有仓库，通过 Trae / VS Code 直接连着读写。
- **知识沉淀对外展示**：引入 **MkDocs**，写一个自动化脚本，每次你有满意的课题产出时，自动抓取 `output/` 里的 Markdown，生成静态网站推送到 GitHub Pages。这样别人只需打开一个网址，就能看到整洁的结构和漂亮的排版。
