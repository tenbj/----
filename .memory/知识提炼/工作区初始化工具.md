## 2026-05-06 · `.memory/.system` 路径退休

**核心观点**：
- `.memory/` 只保存当前态记忆，不再保留 `.system` 镜像目录。
- 初始化工具、SSOT 标准和规范化脚本必须同口径处理：不创建 `.memory/.system/`，发现该目录时按旧残留迁出当前态。

**关键框架/模型**：
- `.system/standards/` 是标准源，历史快照进入 `.history/.system/`。
- `.memory/` 的合法当前态目录只保留 `对话记录/`、`系统记录/`、`知识提炼/`，根文件只保留 `全局知识地图.md`。
- EXE 版本追踪“初始化出的工作区骨架规格”，因此骨架行为变化需要同步 `SKELETON_VERSION` 与 `build.ps1` 输出版本。

**与其他话题的关联**：
- → 项目运作机制：SSOT 标准、rules、skills、脚本和 EXE 模板必须从同一事实源收敛。
- → PowerShell 编码陷阱：模板中的 `.ps1` 同步时要保持 UTF-8 with BOM，避免中文路径和输出乱码。

## 2026-05-06 · 03_代码程序目录版本判定
**核心观点**：
- `03_代码程序` 是代码程序整体版本目录，不应长期停留在 `v0.0.0`；只要目录内已有源码、构建缓存或交付产物，就应进入非零版本管理。
- 规范化/体检脚本不能只统计子目录下的直接文件；对 `03_代码程序` 必须把有效子目录也视为内容，否则 `src/dist/build_tmp` 会被误判为空目录。

**关键框架/模型**：
- 普通 `01/02` 分类目录仍按直接内容文件判断空/非空。
- `03_代码程序` 使用有效内容项判断：除 `.gitkeep` 外，文件和目录都计入内容项；空目录才保持 `v0.0.0`，非空目录保留/要求当前非零版本。
- EXE 内置模板变更属于可交付程序补丁，需要同步 `SKELETON_VERSION`、`build.ps1` 输出版本、代码目录版本和 `dist/` 产物名。

## 2026-05-07 · 初始化工具模板同步闭环

**核心观点**：
- 05 初始化工具的内置骨架不能只看目录是否存在，必须把 live `.agents/rules`、`.agents/skills`、`.system` 与 `src/templates/` 做逐文件哈希对比。
- 发现 live 骨架变更后，应按受管范围全量替换模板；即使某一块当前 0 差异，也可一起覆盖，避免局部同步造成后续漂移。
- 内置模板变更属于 EXE 骨架规格演进，需要同步升级 `SKELETON_VERSION`、`build.ps1` 输出版本、`03_代码程序_v*` 目录版本和 `dist/` 产物名。

**关键框架/模型**：
- 比对口径：忽略 `__pycache__` 等运行缓存，只比较可交付骨架文件。
- 验证闭环：模板哈希 0 差异 → `py_compile` 通过 → PyInstaller 产出新 EXE → 直接调用 `initialize_workspace()` 做无 GUI 烟测。
- 烟测优先走源码函数而不是双击 EXE，可避免 GUI 弹窗阻塞，同时验证核心初始化逻辑。

**与其他话题的关联**：
- → 版本管理：output 子项目先 PROJECT 快照并 bump，知识提炼文件修改前走 MEMORY 备份。
- → 项目骨架演化机制：live rules/skills/system 是当前事实源，EXE 模板是可交付副本，二者必须周期性收敛。
