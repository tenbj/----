# 知识提炼 · 项目运作机制

> 从研究过程中提炼的核心结论与知识点，持续更新。

---

## 2026-05-03 · 项目运作链路

**核心观点**：
- 知识研究工作区由 `.agent`、`output`、`.history`、`.memory` 四层共同构成。
- 子项目是工作边界，备份是安全网，版本是变更语义，记忆是长期上下文。
- 只有把项目内容与系统治理内容分流存储，后续维护才不会越来越乱。

**关键框架/模型**：
- 新话题进入 -> 创建子项目 -> 在子项目内产出文件 -> 修改前备份并升级版本 -> 回复后沉淀记忆
- FILE / CONFIG / FOLDER 三种备份模式分别对应内容文件、规则文件、Skill 文件夹
- PROJECT / SYSTEM / BOTH 三种记忆路由分别对应项目摘要、系统治理摘要和双向写入

**与其他话题的关联**：
- → 版本管理：提供备份流程与版本号演进语义
- → 记忆系统治理方案：提供项目记忆与系统治理记忆的分流边界

---

## 2026-05-03 · 记忆层重设计

**核心观点**：
- `.memory/` 应只保留当前态，不再堆叠多个同主题的 `v*` 文件。
- `.history/.memory/` 应只负责保存历史版本，形成单一且清晰的记忆历史区。
- 旧的整目录快照不应继续混在当前镜像旁边，应被视为 legacy 产物。

**关键框架/模型**：
- 追加型记忆：`.memory/对话记录/`、`.memory/系统记录/`
- 当前态稳定文件：`.memory/知识提炼/{话题名}.md`、`.memory/全局知识地图.md`
- 历史态归档：`.history/.memory/知识提炼/`、`.history/.memory/全局知识地图/`

**与其他话题的关联**：
- → 记忆系统治理方案：把“项目记忆”和“系统治理记忆”的分流，进一步扩展成“当前态 / 历史态”分层

## 2026-05-03 · Output 版本留存规则修正

**核心观点**：
- `output/` 子项目目录不应堆积多个旧版本工作文件，只应保留当前最新版本。
- `output` 的旧版本文件必须连同原版本号一起进入 `.history/output/对应子项目/`。
- 如果无法从旧版本切换到新版本，脚本应直接失败，而不是偷偷复制出新文件并把旧文件留在工作区。

**关键框架/模型**：
- 当前工作区：`output/{子项目}/{文件}_v最新版本.xxx`
- 历史版本区：`.history/output/{子项目}/{文件}_v旧版本.xxx`
- 执行顺序：先复制旧版本到 `.history`，再把工作文件切换到新版本号

**与其他话题的关联**：
- → 版本管理：明确"历史版本归历史区，工作区只留最新态"的版本留存边界

## 2026-05-04 · 记忆链路的断点

**核心观点**：
- 四层记忆（对话记录、系统记录、知识提炼、全局知识地图）在实际执行中，对话记录和知识提炼容易被跳过。
- 根因不是规则缺失，是规则被执行者当成了"任务完成后的额外步骤"。
- 防护需要双层：规则层面的强制条款 + 执行检查清单的自检项。

**关键框架/模型**：

| 断点 | 表现 | 修复 |
|------|------|------|
| 对话记录遗漏 | 任务完成后忘记追加；BOTH 被误判为 SYSTEM | `memory-rules.md` 顶部强制条款；检查清单第 0 项自检；数量自检 |
| 知识提炼遗漏 | Rule 写"必须"但 Skill 写"手动触发"，矛盾给了跳过台阶 | 需统一为"每次 output 文件增删改后自动触发" |
| "补丁/延续"陷阱 | 在已有子项目文件夹中做修改时，不被当成新子项目而跳过记忆 | 禁止行为条款：只要在 output/ 下有独立文件夹就是子项目 |

**与其他话题的关联**：
- → 记忆系统防漏机制：三道防线的完整设计与实施记录

## 2026-05-05 · 子项目三分类结构

**核心观点**：
- 子项目根目录应只保留 `目录.md`、`版本记录.md` 和三个分类子文件夹，内容文件不再平铺在根目录。
- 三个分类子文件夹分别承担问答、研究、代码三类产出：`01_问题答疑_v*`、`02_课题研究_v*`、`03_代码程序_v*`。
- 子文件夹带版本号，内部文件不带 `_v*` 版本号；内部文件用 `{NN}_{标题}.{扩展名}` 表达分类内顺序。
- 体检脚本负责只读发现问题，规范化脚本负责在备份后补齐新结构；老平铺项目不自动搬迁，除非明确更新对应子项目。

**关键框架/模型**：
- 新项目创建：`new_project.ps1` 创建子项目根目录，同时创建三个 `v0.0.0` 空分类文件夹与三节 `目录.md`
- 研究产出路径：结构化研究统一进入 `02_课题研究_v*`
- 版本表达：子项目版本表示整体快照，分类文件夹版本表示分类内容状态，内部文件不再独立携带版本号
- 03 子项目已作为新结构示例：`03_项目运作机制_v1.3.0/02_课题研究_v1.0.0/01_项目运作机制说明.html`

**与其他话题的关联**：
- → 项目骨架演化机制：三分类结构属于工作区骨架持续自我迭代后的新目标形态
- → 版本管理：进一步收敛“版本只在边界层表达”的规则
- → 记忆系统防漏机制：三分类结构让落盘路径和记忆路由更容易被检查
