# 知识提炼 · 项目骨架演化机制

> 从系统治理和工作区初始化工具中提炼出的项目骨架自我迭代模型。

---

## 2026-05-05 · EXE、Skill、治理与迁移闭环

**核心观点**：
- 这个项目的骨架不是静态模板，而是一套可循环升级的知识工作区系统。
- EXE 负责把骨架带给用户；Skill 负责日常落盘、版本、记忆和规范化；00 系统治理负责沉淀系统问题和维护决策；05 工作区初始化工具负责把治理成果重新发布成新版 EXE。
- 老项目升级不应只靠 EXE 覆盖完成。更稳的模型是“新版 EXE 注入新骨架 + 项目规范化 Skill 迁移旧状态”。
- 纠偏：首次剥离/首次初始化不是循环的一部分。它是一次性引导，先把骨架思路通过 05 封装成首版 EXE，再创建标准工作区；后续才进入“提问、落盘、备份、记忆、治理、重新封装、老项目规范化迁移”的循环。

**关键框架**：
1. 首次剥离/初始化：形成项目骨架思路，把 rules、skills、memory、output、history 等骨架资产整理进 05，并封装首版 EXE。
2. 首次使用：用户运行首版 EXE，创建标准工作区骨架，工作区获得 `.agents`、`.memory`、`.history`、`input`、`output` 等基础结构。
3. 后续产出：用户提问触发子项目管理，回答必须进入 output 文档。
4. 留痕：每次修改前由版本控制备份写入 `.history`，每次有效信息增量由记忆管理写入 `.memory`。
5. 治理：系统问题进入 `output/00_系统治理_*`，形成原因、决策、修复方案和维护入口。
6. 发布：维护 AI 将已验证的 rules、skills、scripts、标准同步到 `output/05_工作区初始化工具_*/src/templates`，重新封装 EXE。
7. 迁移：用户在老项目运行新版 EXE 后，再通过项目规范化 Skill 按 `09_项目检查标准` 整理旧框架。
8. 循环：老项目进入新标准后继续提问、落盘、记忆、备份，并在发现新问题时再次进入治理。

**与其他话题的关联**：
- → `系统治理`：00 子项目是维护 AI 的入口和治理记录中心。
- → `工作区初始化工具`：05 子项目是骨架发布器，必须同步模板并重新封装 EXE。
- → `项目检查标准`：09 子项目为老项目规范化提供验收标准。
- → `记忆系统防漏机制`：该闭环依赖对话记录、系统记录、知识提炼和全局知识地图不遗漏。

## 2026-05-05 · 老项目升级的可执行语义

**核心观点**：
- 老项目升级必须拆成两段可验证动作：`初始化工作区_v1.6.0.exe` 负责注入最新骨架资产，`项目规范化 --Fix` 负责迁移旧状态。
- EXE 的职责不是迁移旧内容，而是安全分发骨架：对受管 Rules/Skills 先备份再替换，对 `.memory`、`input`、`output` 只补缺。
- 规范化 Skill 的职责不是生成骨架模板，而是在模板已经注入后整理旧项目：补 `目录.md`、补 `版本记录.md`、重建全局知识地图、处理内容文件编号和历史残留。
- 全局知识地图属于脚本维护对象。新建子项目时应由 `new_project.ps1` 显式 UTF-8 读写并备份，避免 AI 手工整文件写回造成 mojibake。

**关键框架/模型**：
1. 分发阶段：运行新版 EXE，备份旧 `.agents` 受管项，替换为模板中的最新 rules/skills。
2. 迁移阶段：运行 `项目规范化 --Fix`，对旧 output 子项目先快照，再补齐目录、版本记录和编号。
3. 记忆阶段：若全局知识地图缺失、乱码或与 `output/` 不一致，由规范化脚本先备份旧地图，再按当前 `output/` 重建。
4. 验收阶段：再跑 `项目规范化 --Check` 或框架体检，确认新骨架和旧状态已经对齐。

**与其他话题的关联**：
- → `PowerShell 编码陷阱`：所有含中文的 `.ps1` 必须保持 UTF-8 BOM，脚本内部读写 Markdown 要用显式 UTF-8 API。
- → `工作区初始化工具`：v1.6.0 是首次把“老项目安全升级”语义写进 EXE 源码和构建产物的版本。
