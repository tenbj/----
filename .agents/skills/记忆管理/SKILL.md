# 记忆管理技能（Memory Management）

## 四类记忆文件

| 文件 | 路径 | 触发时机 | 写入方式 |
|------|------|---------|---------|
| **项目对话记录** | `.memory/对话记录/{子项目文件夹名}.md` | 按信息增量判断（见下方"核心触发规则"） | 追加（无需版本备份） |
| **系统治理记录** | `.memory/系统记录/` 下按分类写入 | 按信息增量判断（见下方） | 追加到对应分类文件（无需版本备份），同一事件涉多维度时分别写 |
| **知识提炼** | `.memory/知识提炼/{话题名}.md` | 每次 output/ 文件增删改后自动触发（见下方触发条件） | 当前态稳定文件名，历史版本进入 `.history/.memory/知识提炼/` |
| **全局知识地图** | `.memory/全局知识地图.md` | 每次新建子项目时 | 当前态稳定文件名，历史版本进入 `.history/.memory/全局知识地图/` |

> `{子项目文件夹名}` 使用带时间戳的工作边界名，例如：`减肥第一性原理_202605031505`。
> `{话题名}` 使用稳定主题名，不带时间戳和版本号，例如：`减肥第一性原理`。

---

## 第 0 步：判断记录路由

在写入任何记忆之前，先判断本轮对话属于哪一种：

- `PROJECT`：研究主题本身、业务分析、研究结论、输出文件内容
- `SYSTEM`：Rule、Skill、脚本、备份策略、目录规范、Memory 设计、自动化工作流
- `BOTH`：同时包含项目内容与系统治理内容

执行要求：

1. `PROJECT`：只写 `.memory/对话记录/{子项目文件夹名}.md`
2. `SYSTEM`：按内容分类写入：
   - Rule 增删改 → `.memory/系统记录/规则变更记录.md`
   - Skill 增删改 → `.memory/系统记录/技能变更记录.md`
   - 脚本 bug/编码/编译 → `.memory/系统记录/脚本治理记录.md`
   - 跨领域教训/根因/纠偏 → `.memory/系统记录/教训库.md`
   同一事件涉多维度时分别写入对应文件，完成后更新 `.memory/系统记录/索引.md` 中各文件条目计数
3. `BOTH`：分别写两边，禁止合并成一条摘要

## 核心触发规则：按信息增量判断，而非话题大小

记忆记录的触发条件**不是"做完一个大任务"，而是"本轮回复产生了值得记录的信息"**。

**必须记录的情况**（满足任一即触发）：
- AI 输出了决策、分析结论、优化方案、设计思路（**包括聊天中的知识回答**，不限于文件产出）
- AI 执行了操作步骤（修改文件、创建目录、执行命令）
- 对话中暴露了经验教训、踩坑记录、纠偏规则
- 用户明确要求记录

**无需记录的情况**：
- 纯信息确认（"是/否"、"好的"）
- 问候语、闲聊
- 重复已记录过的内容（无新增信息）
- `目录.md` 的同步维护（它是子项目文件增删改的附带操作，文件操作本身已触发记录）

**执行方式**：
- 每轮回复结束时，先完成记忆写入，再发送最终回复
- 记忆写入不是"任务完成后的额外步骤"，而是回复本身的一部分
- 使用 TodoWrite 时，最后一个 todo 固定为"写入记忆"，确保不会遗漏

## 文件 A：项目对话记录

### 路径规则
```
.memory/对话记录/{子项目文件夹名}.md
```
例：`.memory/对话记录/减肥第一性原理_202605031505.md`

### 写入规则
- **无版本号**，文件名固定不变
- **追加写入**，不覆盖，不备份到 `.history`
- 每次 AI 回复后，若路由为 `PROJECT` 或 `BOTH`，在文件末尾追加本轮摘要
- 若文件不存在，先按初始化模板创建，再执行追加
- 若本轮是纯系统治理对话，禁止写入该文件

### 追加格式
```markdown
## {YYYY-MM-DD HH:MM}

**用户问**：{用户问题的一句话概括}
**AI做了**：{本次回复的核心操作，一句话}
**产出文件**：{如有，列出文件路径；无则省略}

---
```

### 初始化模板（首次创建时）
```markdown
# 对话记录 · {话题名}

> 记录本话题所有对话的关键摘要，按时间追加。

---

## {YYYY-MM-DD HH:MM}

**用户问**：...
**AI做了**：...

---
```

## 文件 B：系统治理记录

### 路径规则
```
.memory/系统记录/
├── 规则变更记录.md
├── 技能变更记录.md
├── 脚本治理记录.md
├── 教训库.md
└── 索引.md
```

系统治理记录不再写入旧聚合文件 `rules-skills.md`。该文件只作为旧版迁移源：发现时应先读取内容，按分类追加到上面四个正文文件，更新 `索引.md`，再归档原文件。

### 写入规则
- **无版本号**，五个文件名固定不变
- **追加写入**，不覆盖，不备份到 `.history`
- 每次 AI 回复后，若路由为 `SYSTEM` 或 `BOTH`，按内容维度写入一个或多个正文文件
- `索引.md` 只做入口与条目计数，不写入记录正文
- 若正文文件不存在，先按初始化模板创建，再执行追加
- 纯系统治理对话禁止回写到任何子项目对话记录

### 分类规则

| 记录内容涉及 | 写入文件 |
|------------|---------|
| Rule 文件增删改 | `规则变更记录.md` |
| Skill 文件增删改 | `技能变更记录.md` |
| 脚本 bug、编码、编译、封装 | `脚本治理记录.md` |
| 跨领域教训、根因分析、纠偏规则 | `教训库.md` |

同一事件触达多个维度时分别写入对应文件。比如“修改 Skill，同时修复 ps1 编码事故，并沉淀以后避免乱码的原则”，需要同时写 `技能变更记录.md`、`脚本治理记录.md`、`教训库.md`。

### 追加格式
```markdown
## {YYYY-MM-DD HH:MM}

**用户问**：{本轮系统治理问题的一句话概括}
**AI做了**：{本轮系统治理操作，一句话}
**产出文件**：{如有，列出文件路径；无则省略}

---
```

### 初始化模板（首次创建时）
```markdown
# 系统记录 · {分类名}

> {分类职责说明}，按时间追加。

---

## {YYYY-MM-DD HH:MM}

**用户问**：...
**AI做了**：...

---
```

### 执行检查清单
每次准备结束当前回复前，都必须完成以下检查：

**第 0 项（最优先）：本轮是否输出了知识性回答？**
- 是 → 不管是否改了文件、是否调了其他 Skill，都必须写入对应子项目的对话记录
- 是但找不到对应子项目 → 话题归属漏了，按 direction-rules.md 强制门补建子项目，再写记录
- 否（纯确认/闲聊）→ 继续检查第 0.5 项

**第 0.5 项：本轮是否产生了信息增量？**
- 对着"核心触发规则"的 4 条"必须记录"，逐条核对本轮回复
- 只要命中任意一条 → 立即写入记忆，再发最终回复
- 不要自我合理化跳过（如"这只是个小问题""等会儿一起记"）

1. 判断本轮路由是 `PROJECT`、`SYSTEM` 还是 `BOTH`
   - **关键规则：只要涉及 output/ 下文件的增删改，至少包含 PROJECT。讨论系统机制 + 产出文件 ≥ BOTH。**
   - **关键规则：不要以"这不是新话题/只是补丁/只是修改"为由判 SYSTEM。看在哪个子项目文件夹里做。**
2. 若涉及项目内容，定位 `.memory/对话记录/{子项目文件夹名}.md`
3. 若涉及系统内容，按分类定位 `.memory/系统记录/` 下的正文文件；同一事件可写多个分类文件
4. 若目标文件不存在，先初始化
5. **先写入记忆，再发送回复**（顺序不可颠倒）
6. **自检：`.memory/对话记录/` 下的文件数 ≥ `output/` 下的子项目文件夹数？数量不对应立即补建。**

## 文件 C：知识提炼

### 路径规则
```
.memory/知识提炼/{话题名}.md
```
例：`.memory/知识提炼/减肥第一性原理.md`

### 写入规则
- **当前态文件名稳定**，不带版本号，不带时间戳，不在 `.memory/知识提炼/` 中堆多个 `*_v*.md` 文件
- 修改前执行版本控制备份（Skill：版本控制备份，`MEMORY` 模式）
- 历史版本统一进入 `.history/.memory/知识提炼/`

### 触发条件（自动触发）

知识提炼不再是"手动触发"。以下任一情况发生时，必须执行知识提炼：

**触发条件 A：output 文件有实质性内容变更**
- 新建子项目输出文件（含核心结论、分析框架等实质性内容）
- 对已有输出文件做 MAJOR 或 MINOR 级别修改（新增章节、新增结论、框架调整）
- **不触发**：PATCH 级别修改（改错别字、调整格式、修正笔误）

**触发条件 B：系统治理变更产生可复用认知**
- Rule/Skill/脚本修改的过程中，暴露出新的架构性问题或发现了新的设计模式
- 修改原因和修复方案本身值得作为"知识"长期保留

**触发条件 C：用户明确要求**
- 用户直接说"提炼一下"、"沉淀知识"等

**执行要求**：
- 若触发条件 A 或 C → 写 `.memory/知识提炼/{话题名}.md`（话题名 = 子项目主题名，不带时间戳）
- 若触发条件 B → 判断是否已有对应话题文件，有则更新，无则新建
- 版本升级规则：新增章节/框架 → MINOR；修正已有结论 → PATCH
- 当前版本号写入文件头隐藏注释：`<!-- memory-version: x.y.z -->`
- 若发现 `.memory/知识提炼/` 中已有 `*_v*.md`，视为历史副本误留在当前区：先确认 `.history/.memory/知识提炼/` 是否已有同名/同内容备份，再整理出 `.memory/知识提炼/`

### 内容模板
```markdown
<!-- memory-version: 1.0.0 -->
# 知识提炼 · {话题名}

> 从研究过程中提炼的核心结论与知识点，持续更新。

---

## {YYYY-MM-DD} · {本次提炼的主题}

**核心观点**：
- ...

**关键框架/模型**：
- ...

**与其他话题的关联**：
- → {话题名}：{关联描述}

---
```

## 文件 D：全局知识地图

### 路径规则
```
.memory/全局知识地图.md
```

### 写入规则
- **当前态文件名稳定**，固定使用 `.memory/全局知识地图.md`
- 修改前执行版本控制备份（Skill：版本控制备份，`MEMORY` 模式，通常使用 `MINOR`）
- 历史版本统一进入 `.history/.memory/全局知识地图/`
- 每次新建子项目时由**子项目管理 Skill**自动触发更新
- 在表格中追加一行
- 当前版本号写入文件头隐藏注释：`<!-- memory-version: x.y.z -->`

### 内容模板
```markdown
<!-- memory-version: 1.0.0 -->
# 全局知识地图

> 所有研究子项目的总索引，新建子项目时自动更新。

| 话题 | 子项目文件夹 | 创建时间 | 状态 | 核心结论（一句话） |
|------|------------|---------|------|----------------|
| 减肥第一性原理 | 减肥第一性原理_202605031505 | 2026-05-03 | 进行中 | - |
```

## 禁止行为
- ❌ 对话记录使用覆盖写入（必须追加）
- ❌ 纯系统治理对话写入子项目对话记录
- ❌ 知识提炼跳过 `MEMORY` 备份直接修改
- ❌ 在 .memory/ 下按项目建子文件夹（必须按类型分）
- ❌ 全局知识地图不在新建子项目时更新
- ❌ 将 `BOTH` 路由混写成一条，不做拆分
- ❌ 将新系统治理记录继续写入旧 `rules-skills.md`
- ❌ 发现旧 `rules-skills.md` 时不迁移内容、只把文件当多余项搬走
- ❌ 在 `.memory/知识提炼/` 当前目录中保留多个 `*_v*.md`
- ❌ 在 `.memory/` 当前目录中保留多个 `全局知识地图_v*.md`
- ❌ 对 `.memory/` 或其子目录做整目录快照
