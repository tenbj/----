# 严格规范化流程

## 总体原则

严格规范化分为两阶段：

1. 脚本阶段：执行确定性检查和机械修复。
2. 大模型阶段：拿标准做语义自查，处理脚本无法覆盖或覆盖不充分的差异。

脚本阶段完成后不能直接结束。必须由大模型复核 `output/`、`.history/`、`.memory/`、`.system/` 是否与 `.system/standards/` 的标准一致。

## 标准读取顺序

每次执行前先读：

1. `.system/standards/workspace-spec.json`
2. `.system/standards/工作区骨架规格.md`
3. `.system/standards/工作区命名规范.md`

若三者存在差异，以 `workspace-spec.json` 为机器可读 SSOT；人类可读标准用于解释意图。若发现标准之间互相矛盾，先记录到 `00_系统治理`，不要靠猜测悄悄改标准。

## 脚本阶段

只读检查：

```powershell
powershell -ExecutionPolicy Bypass -File "e:\my_project\知识研究\.agents\skills\项目规范化\scripts\normalize.ps1" --Check
```

修复检查：

```powershell
powershell -ExecutionPolicy Bypass -File "e:\my_project\知识研究\.agents\skills\项目规范化\scripts\normalize.ps1" --Fix
```

脚本职责：

- 补建缺失根目录和必备目录
- 移走明显多余项
- 检查 `.agents` 必备 rules/skills
- 检查 `output/` 子项目命名与基础内部结构
- 修复 `.memory` 旧聚合记录、知识提炼历史副本、全局知识地图对齐问题
- 检查 `.history` 允许目录和历史快照命名
- 检查 `.system/standards` 必备文件

## 大模型阶段

脚本完成后，大模型必须：

1. 对照标准列出脚本未覆盖、覆盖不足或结果仍不一致的区域。
2. 对 `output/` 子项目做内容理解和分类迁移，不能把平铺老项目当合规终态。
3. 对 `.history/` 做结构和命名自查，确认历史区只承担历史职责。
4. 对 `.memory/` 做当前态洁净度、自身结构、对话记录与 `output/` 对齐检查。
5. 对 `.system/` 做核心骨架存在性检查，不回写标准以迁就当前状态。
6. 能确定修法时直接修；不能确定时记录异常。

## 修改前备份

大模型阶段只要修改文件或移动结构，必须先按目标类型执行 `版本控制备份`：

| 目标 | 备份模式 |
|------|----------|
| `output/` 子项目 | PROJECT |
| `.agents/skills/` | FOLDER |
| `.agents/rules/` | CONFIG |
| `.system/` 子文件夹 | FOLDER |
| `.memory/知识提炼/*.md`、`.memory/全局知识地图.md` | MEMORY |
| `.memory/对话记录/`、`.memory/系统记录/` | 追加型记录，不做版本备份 |

## 结束条件

一次严格规范化结束时必须满足：

- `normalize.ps1 --Check` 已重新运行
- 剩余 FAIL/WARN 要么已修复，要么已记录在 `00_系统治理`
- `output/`、`.history/`、`.memory/` 都经过大模型标准自查
- 修改过的子项目 `目录.md` 与 `版本记录.md` 已同步
- 系统治理记录已写入 `.memory/系统记录/`
